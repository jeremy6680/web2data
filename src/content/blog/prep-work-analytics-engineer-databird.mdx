Avant de démarrer la formation Analytics Engineer chez Databird, il y a un prep-work à faire. Des modules bien pensés pour ceux qui n'auraient pas forcément déjà mis les pieds dans le monde de la data. Certes, je connaissais Git depuis longtemps, mais pour le reste, ces modules m'ont appris des choses concrètes — notamment sur la façon dont un projet data se structure en entreprise, et sur BigQuery.

C'est ce dont j'ai envie de parler ici.

---

## Git : rien de nouveau, mais une bonne remise à plat

Le module Git était clairement conçu pour des gens qui n'ont jamais versionné de code. Initialiser un dépôt, créer une branche, faire une pull request, gérer un conflit… Rien que je ne faisais pas déjà au quotidien avec Bitbucket.

Mais ça m'a quand même rappelé quelque chose d'important : Git, c'est une des compétences les plus transversales qui soit. En web, on l'utilise pour versionner du code PHP ou du JavaScript. En data, on l'utilise pour versionner des requêtes SQL, des modèles dbt, des notebooks Python. Le paradigme est exactement le même. Et les bonnes pratiques aussi : des commits clairs, des branches pour isoler le travail, des pull requests pour documenter les changements.

Si t'es dev web et que tu arrives en data, Git c'est ton atout direct. Ne le sous-estime pas.

## Projets data en entreprise : un cadre que je ne connaissais pas

C'est là que ça devient vraiment intéressant pour moi. Ce module m'a donné une vision structurée de ce à quoi ressemble un projet data côté organisation — quelque chose qu'on n'apprend pas en développant des sites WordPress.

### La maturité data : un concept qui n'existe pas vraiment en web

En web, on parle rarement de "maturité" d'une entreprise par rapport à son site. Soit ils ont un site, soit ils n'en ont pas. Soit il est bien fait, soit il est moyen. Mais il n'y a pas vraiment de grille d'évaluation formalisée.

En data, c'est différent. Le cours présente un framework en **cinq piliers** pour évaluer où en est une entreprise :

- **Infrastructure** : quels outils pour collecter et stocker les données ?
- **Personnes & culture** : est-ce que la data est quelque chose que tout le monde comprend et utilise, ou c'est réservé à deux personnes dans un coin ?
- **Outils** : Python, SQL, Tableau, Power BI — qu'est-ce qui est utilisé, et est-ce standardisé ?
- **Organisation** : est-ce qu'il y a un pôle data centralisé, ou des gens éparpillés dans chaque département ?
- **Processus** : comment les équipes non-techniques communiquent avec les équipes data ? Ticketing, wikis, sessions Q&A…

Et à partir de là, on peut situer une entreprise sur une échelle à quatre niveaux : **Data Reactive** (fichiers Excel éparpillés, zéro culture data), **Data Scaling**, **Data Progressive**, jusqu'à **Data Fluent** (la maturité maximale, où la data est intégrée dans toutes les décisions).

Ce que j'y retrouve côté web : la notion d'**organisation** m'est familière. Dans une agence, on a aussi des processus plus ou moins formalisés, des outils plus ou moins standardisés, des équipes plus ou moins alignées. Mais le fait de l'avoir modélisé de façon aussi claire, c'est quelque chose que je n'avais jamais vu appliqué au web.

### La roadmap d'un projet data vs un projet web

En web, une livraison ressemble souvent à ça : brief client → maquette → développement → recette → mise en ligne. C'est linéaire, même quand on prétend faire de l'agile.

Un projet data suit à peu près le même chemin, mais avec des étapes différentes :

1. **Cadrage** — définir la problématique, identifier les KPI, organiser l'équipe
2. **Récolte des données** — accéder au data warehouse, tester les requêtes SQL, identifier les manques
3. **Analyse** — nettoyage, exploration (EDA), analyses statistiques ou prédictives
4. **Restitution** — dashboards, storytelling, recommandations

Ce qui m'a frappé : le cadrage représente environ 20 % du temps total. Et c'est là que tout se joue. En web, on saute souvent directement dans le code parce que "le brief est clair". En data, si tu pars sans avoir bien compris le besoin métier, tu vas produire une analyse qui ne sert à rien. La méthode des **5 Pourquoi** (5 Why) est présentée comme outil pour creuser jusqu'à la vraie cause d'un problème — simple, efficace, et applicable partout, en data comme en web.

La méthode **MoSCoW** (Must Have / Should Have / Could Have / Will Not Have) est aussi abordée pour prioriser les tâches. Je l'avais déjà croisée en gestion de projet web, mais rarement appliquée aussi rigoureusement.

### Les méthodes agiles : un terrain connu, des nuances nouvelles

Scrum, Kanban, sprints, user stories… Le vocabulaire agile n'est pas étranger à un développeur web. Ce qui m'a semblé différent ici, c'est l'**insistance sur la collaboration avec les équipes métier** dès le début du projet. En data, si tu produis une analyse sans impliquer le business, tu risques de répondre à la mauvaise question. L'itération ne sert pas à livrer du code, elle sert à valider des hypothèses.

## BigQuery : mon premier vrai contact avec un data warehouse

Là, on entre dans du nouveau territoire pour moi.

### GCP et BigQuery, c'est quoi concrètement ?

**Google Cloud Platform (GCP)** est une suite de services cloud. Parmi eux, **BigQuery** est ce qu'on appelle un **data warehouse** — un entrepôt qui stocke des données structurées et permet d'exécuter des requêtes SQL sur des volumes massifs, sans avoir besoin de gérer l'infrastructure.

Concrètement, dans la **Modern Data Stack**, BigQuery joue le rôle de colonne vertébrale : les données arrivent de sources variées (marketing, logistique, CRM…), elles sont centralisées dans un **data lake** (par exemple Google Cloud Storage), puis transformées et chargées dans BigQuery pour être analysées. Des outils comme **dbt** interviennent pour transformer les données, et **Power BI** ou **Looker** pour les visualiser.

```
Sources de données → Data Lake (Cloud Storage) → BigQuery (Data Warehouse) → Visualisation (Looker, Power BI)
```

C'est l'architecture ELT (Extract, Load, Transform) — à l'opposé de l'ETL classique où on transforme avant de charger. En pratique, ça veut dire qu'on charge les données brutes d'abord, et on les transforme ensuite directement dans BigQuery, en SQL.

### La structure des données dans BigQuery

BigQuery organise les données en trois niveaux :

- **Projet** → le niveau le plus haut (par exemple : `mon-projet-spotify`)
- **Dataset** → un groupe de tables (par exemple : `dataset_marketing`)
- **Table** → les données elles-mêmes

Et deux concepts qu'il faut distinguer :

- **Data Lake** : stockage brut, non transformé, toutes sources confondues
- **Data Mart** : sous-ensemble du data warehouse, dédié à un département ou une fonction précise (finance, marketing, logistique…)

Ce qui m'a marqué : l'importance des **conventions de nommage**. Exactement comme en web — si tu travailles seul ça va, mais en équipe, sans conventions, c'est le chaos. En BigQuery, on utilise des identifiants uniquement composés de lettres, chiffres et underscores. Pas de caractères spéciaux, des noms clairs et cohérents.

### IAM, rôles et permissions : un parallèle avec WordPress

La gestion des accès dans BigQuery passe par le système **IAM** (Identity and Access Management) de Google Cloud. On crée des comptes de service, on attribue des rôles (Data Viewer, Data Editor, BigQuery Admin…), et on génère des clés pour permettre à des outils externes — comme Power BI — de se connecter.

Ça m'a rappelé la gestion des rôles dans WordPress : Subscriber, Editor, Administrator… Le principe est exactement le même. Ce qui change, c'est l'enjeu : dans BigQuery, on parle de données potentiellement sensibles et de coûts liés aux requêtes. Donc la gestion des accès est vraiment critique.

### La tarification à l'usage

Autre concept nouveau pour moi : **BigQuery ne facture pas un abonnement fixe, mais l'usage**. Trois dimensions :

- Le stockage (court terme vs long terme)
- Le volume de données analysé par les requêtes
- Le streaming (déplacement de données en temps réel)

Ça change complètement la façon d'écrire ses requêtes. En web, si une requête SQL est lente, c'est un problème de performance. Dans BigQuery, si une requête est inefficace, c'est aussi un problème de coût. D'où l'importance de techniques comme le **partitionnement** et le **clustering** des tables, pour limiter la quantité de données scannées à chaque requête.

## Ce que je retiens globalement

Ce prep-work m'a confirmé quelque chose que je pressentais : le monde de la data a ses propres conventions, ses propres outils, ses propres façons de structurer les projets — mais beaucoup de ses fondamentaux sont les mêmes que dans le développement web.

Les points communs sont réels : Git, les méthodes agiles, la gestion de projet, l'importance de la documentation, les conventions de nommage, la gestion des rôles et accès. Si tu viens du web, tu n'arrives pas les mains vides.

Mais les différences comptent aussi : la notion de maturité data, l'architecture ELT, la facturation à l'usage, le rôle central du métier dans la validation des analyses… C'est un état d'esprit différent. On ne livre pas du code, on produit des insights. Et un insight qui répond à la mauvaise question, ça ne vaut rien.

La suite, c'est d'aller beaucoup plus loin dans tout ça. Mais pour un point de départ, ce prep-work pose des bases solides — et donne envie de continuer.
